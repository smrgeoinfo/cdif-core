# AGENTS.md - AI Agent Guidance for CDIF Core

## Project context

This repository hosts the CDIF Core profile — the base level of CDIF conformance for metadata records describing scientific datasets. The Core profile defines mandatory and optional properties using schema.org vocabulary, with JSON-LD as the serialization format.

## Key files

- `CDIFCoreClasses.md` — Human-readable specification of all classes and properties
- `cdifCoreStructuredSchema.json` — JSON Schema for validation (generated from metadataBuildingBlocks)
- `rules.shacl` — SHACL shapes (copied from metadataBuildingBlocks, keep in sync)
- `examples/` — Validated JSON-LD examples from diverse sources

## Example sources and conventions

Examples come from multiple sources and are converted to CDIF format:

- **Kaggle, NCEI, Copernicus, PANGAEA** — harvested from landing page JSON-LD using `geocodes_harvester.py`
- **Dataverse** (Harvard, Borealis, DANS, JHU) — via Dataverse schema.org export API (`/api/datasets/export?exporter=schema.org&persistentId={doi}`)
- **PSDI** — converted from DCAT using `DCAT/dcat_to_cdif.py`
- **ADA** — stripped to Core-only properties from full test records

### Classification rule

Examples with `spatialCoverage`, `temporalCoverage`, or `variableMeasured` belong in the **Discovery** repo, not here. Core-only examples go here.

### CDIF conversion requirements

When adding examples, ensure:

1. `@context` has explicit prefix declarations (`schema`, `dcterms`, `dcat`, `prov`) — never `@vocab`
2. `schema:` prefix on all schema.org property names
3. `@type` as arrays (e.g., `["schema:Dataset"]`)
4. `schema:subjectOf` with `dcat:CatalogRecord`, `dcterms:conformsTo` including `https://w3id.org/cdif/core/1.0`
5. `schema:creator` uses `@list` wrapper for author ordering
6. Namespace `http://schema.org/` (never `https://`)
7. **Never strip extra properties** — open-world assumption means unknown properties are valid
8. If source has both `author` and `creator` with identical content, remove `author` (CDIF recommends `creator`)
9. Unprefixed properties matching schema.org names → assume `schema:` and note assumption in `subjectOf/description`
10. Truly unknown unprefixed properties → assign to `unk:` (`https://ex.org/unknown/`) namespace

### Synced files

These files are copied from the metadataBuildingBlocks repo and should be updated when the source changes:

- `rules.shacl` ← `metadataBuildingBlocks/_sources/cdifProperties/cdifCore/rules.shacl`
- `cdifCoreStructuredSchema.json` ← generated by `tools/resolve_schema.py --structured`

## Validation

```bash
# Validate an example against the Core schema
python -c "
import json
from jsonschema import Draft202012Validator
schema = json.load(open('cdifCoreStructuredSchema.json'))
doc = json.load(open('examples/kaggle-iris.jsonld'))
errors = list(Draft202012Validator(schema).iter_errors(doc))
print(f'{len(errors)} errors')
"
```

## Working with the ODIS archive

The `ODIS/` directory contains archived type-specific templates and examples from the original Ocean Data and Information System schema.org guidance (Person, Organization, Event, CreativeWork, etc.). These predate the CDIF building blocks architecture and use `@vocab` style contexts. They are preserved for reference but are not CDIF-conformant.
